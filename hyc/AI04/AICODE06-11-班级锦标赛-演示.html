核心概念认知层级

概念 学生类比 认知层级
数据驱动设计 游戏皮肤系统——换皮肤不需要重新下载整个游戏 理解层
硬编码 把菜单直接印在餐厅墙上——想加菜要重新刷墙 识别层
roles 系统 班级花名册——来了新同学只需加一行 理解层
增量需求 在已有房子里加一个房间——不是推倒重建 应用层
路演设计决策 苹果发布会不说「有摄像头」，而说「为什么改变了拍照方式」 应用层

典型误概念

M1

加新角色需要改代码

误概念：每次加角色都要修改程序代码

正确认知：数据驱动设计——加一个 json 文件 = 加一个角色，代码不用改

M2

路演就是说「我做了什么功能」

误概念：路演是功能列表的罗列

正确认知：路演要说「我为什么这样设计」——设计决策比功能列表更有价值

M3

属性随便分的角色不可能赢

误概念：只有认真分配的属性才能赢

正确认知：「奇怪」的属性分配有时会产生意外的克制效果

M4

输了说明自己设计失败了

误概念：输了 = 设计失败

正确认知：输了是迭代的起点；「如果重来属性怎么分」才是最有价值的问题

教学准备

教师备课必做（课前）

  • 收集角色文件 — 提前向每位学生收集角色名.png + 角色名.json，放入教师电脑 roles/ 文件夹
  • 测试 roles 系统 — 在教师电脑上跑一遍生成流程，确保角色选择界面正常读取所有角色
  • 准备花名册投屏 — 一个展示全班角色名字 + 特技的页面或截图
  • 准备对阵表 — 单淘汰制，根据班级人数提前画好括号
  • 备用方案 — 如果 roles 系统生成失败，手动在代码里加角色选择下拉菜单

// 硬编码方式（现在） const player = { name: '章鱼怪', hp: 80, atk: 15, def: 5, spd: 8 }; // → 想加角色：改代码 // 数据驱动方式（今天要做） // 读取 roles/角色A.json → 代码只负责读取，不存储数据 // → 想加角色：在 roles/ 文件夹里加一个新 json 文件，代码不变

第二幕：建构 (Construct) — 65分钟

分段一：roles 系统——数据驱动设计实战（20分钟）

1 讲解与演示

  • 展示增量需求文档
  • 讲解四条核心要求
  • 演示数据驱动 vs 硬编码对比

2 窗口B审核

  • 用 Kimi 审核需求文档
  • 找出漏洞和没说清楚的地方
  • 补充答案到需求文档

3 窗口C执行

  • 使用保底提示词
  • 只输出新增/修改的代码
  • 避免 AI 重写整个游戏

分段二：班级锦标赛——全班大乱斗（30分钟）

锦标赛规则

  • 单淘汰赛制
  • 每场对战：一方选自己的角色，另一方选自己的角色
  • 战斗 5 回合，5 回合后谁 HP 多谁赢
  • 如果有一方 HP 归零，提前结束
  • 每场大约 2-3 分钟

🎮 战斗进行中

教师实时解说：回合数、伤害值、特技触发、形势分析

🏆 战斗结束后

分析赢的原因，引导输方思考「如果重来属性怎么分会更好」

第三幕：反思 (Contemplate) — 15分钟

每人完成 3 分钟路演，训练「说设计决策」而非「说功能列表」

1

展示角色（30秒）

「我的角色叫 XX，是一个 XX 型，属性分配是……」

2

设计意图（30秒）

「我选这个打法是因为……我觉得用 XX 特技可以克制……」

3

精彩瞬间（60秒）

「最让我印象深刻的是和 XX 的对战，那一场……」

4

设计复盘（60秒）

「如果重来，我会把 XX 调低一点，把 XX 调高……因为我发现……」

AI 助教使用指南

A 窗口A - 需求文档

增量需求文档模板，包含四条核心要求：

  • 从 roles/ 文件夹读取 *.json
  • 显示角色选择界面
  • 选择确认与战斗加载
  • 不修改原战斗逻辑

C 窗口C - 保底提示词

我已有一个 Phaser.js 战斗游戏...
现在我要加角色选择系统...
只输出新增/修改的代码部分，
不要重写整个游戏。

路演辅助提示词

帮我用 3 个问题引导我做「设计复盘」——只问问题，不给答案。

5分钟日常AI挑战

角色 2.0 版本

给你的角色设计一个新版本

1. 你改了哪些属性，改成了什么数字
2. 这次你想克制哪种打法
3. 你觉得这样改会带来什么新的弱点

不需要跑代码验证，只需要写出「设计意图」
